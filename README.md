# Nexacore Inventory

Modular monolith inventory management backend with React frontend.

## Current Architecture

- Frontend: React + Vite ([frontend](frontend))
- Backend: Spring Boot modular monolith ([src/main/java/com/nexacore/inventory](src/main/java/com/nexacore/inventory))
- Database: PostgreSQL + Flyway migrations
- Deployment: Docker Compose

## Backend Module Layout

- `common/` shared config + global exception handling
- `auth/` JWT login flow and auth DTOs
- `products/` product CRUD domain
- `stock/` stock adjustment business logic (served via `/api/v1/inventory/*` endpoints)
- `integrations/` external inbound API (`/api/v1/integrations/stock-update`)
- `reporting/` reporting endpoints

Each module keeps its own controller/service/repository/dto where applicable.

## Directory Meaning

- `src/` = source code and resources that you edit and commit.
	- `src/main/java/` = Java application code.
	- `src/main/resources/` = YAML, SQL migrations, static config.
	- `src/test/` = automated tests.
- `target/` = build output generated by Maven.
	- compiled classes, packaged JARs, copied resources.
	- safe to delete; Maven recreates it on the next build.

## API Quick Links

- Swagger UI: http://localhost:8080/swagger-ui.html
- Auth login: `POST /api/v1/auth/login`
- Products: `/api/v1/products`
- Inventory stock adjustments: `PATCH /api/v1/inventory/stock-adjustments`
- POS/webhook stock events: `POST /api/v1/integrations/stock-events`
- Stock reconciliation: `POST /api/v1/integrations/reconcile-stock`

## Stock Event Contract

External systems should send:

```json
{
	"sourceSystem": "POS-A",
	"eventId": "evt-10001",
	"eventType": "SALE",
	"sku": "SKU-001",
	"quantity": 2,
	"occurredAt": "2026-04-24T14:30:00+05:30"
}
```

Response format includes:
- `status` (`updated` or `duplicate`)
- `duplicate` boolean
- `eventId`, `sourceSystem`, `eventType`, `sku`
- `beforeStock`, `afterStock`, `processedAt`

Error format is standardized with:
- timestamp
- status
- error
- message
- path

## Notes for Future Microservices Migration

- Keep external API contracts under `/api/v1/*` stable.
- Avoid cross-module direct data access; go through services.
- Extract modules one by one (recommended order: `integrations` or `stock` first).
- Introduce gateway and per-service database only when scaling requires it.
